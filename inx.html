Who’s who in a NERDS app?

Pavel Machuca-Zavarzin
Jan 2

This post aims to take a quick and dirty look at a NERDS (Not the same as NERD) full stack application’s skeleton: Most files’ (usual) locations and functions within the application.

What you’ll see in this post:

  • A wider sweep across a full stack application to see what library is used where. An analogy could be that this is a tracing of a map rather than instructions on how to travel the territories it represents. This is useful for me, hopefully to you as well
  • A few small code samples
  • Links to the repository on which I’m basing this post

What’s a NERDS stack?

Node is a run-time environment possessed of everything needed to execute a Javascript program.

Express is a web app and API framework for Node

React is a front-end JS library to build user interfaces (UIs)

Databases using…

SQL (Structured Query Language) are relational databases. Meaning both storing models and being able to create associations between them

Configuration files at the root level

.babelrc settings that make your code compatible with older browsers
.editorconfig keeps a standard format for the whole project
.eslintrc.json settings for your linter (which gives suggestions and warnings across files)
.github contains information on the repository (commits, branches, remotes, etc)
.gitignore list of files and directories for the version manager (in this case git) to ignore in the repository. This usually means API keys, secrets, and other information to keep private.
.prettierignore list of files and directories for prettier to ignore
.prettierrc.yml configuration for prettier (serves to keep a standard format across files, useful to minimize merge conflicts and keep code uniform)
.travis.yml configuration for Travis (for continuous integration). Meaning that your project in the cloud will automatically update to the latest commit on your repository
package-lock.json is automatically generated as you install your dependencies
package.json lists the project’s dependencies and custom scripts
webpack.config.json configuration for webpack

BACK END

Short Story: Node, Express (routing and error-handling), database models, back end socket, authentication.

Longer Story: Let’s start at:

Back end entry point

  • /server/index.js
  • Imports some libraries described below, and database; specifies port number
  • If the app is not running in production mode, then it imports secrets locally.
  • The current user’s identity is authenticated through passport.js
  • Logging middleware (i.e. morgan): to make the terminal easier to read
  • Body parsing middleware: gives you a req.body (to handle as you like) from the body portion of all requests made to the server
  • Routes to authentication and api (assuming front-end routing will happen with React)
  • Serves static files (usually held at /public, like index.html, styles.css, and bundled js files)
  • ‘Not found’ or 404 handler
const app = require('express')();
const path = require('path');
...
app.use((req, res, next) => {
    if (path.extname(req.path).length) {
      const err = new Error('Not found')
      err.status = 404
      next(err)
    } else {
      next()
    }
  })
  • Error handler
app.use((err, req, res, next) => {
    console.error(err)
    console.error(err.stack)
    res.status(err.status || 500).send(err.message || 'Internal server error.')
})
  • Initializes socket.io
  • Syncs to the database
  • Listens on the specified port number.

This Leads to 4 branches: Database, API, Socket, Authentication

Database

  • /server/db/
  • Contains models and defines their relations
  • Connects to a test database if the app is not being run in production mode

API

  • /server/api
  • Has its own 404 handler
  • Defines requests that can be made to the database
  • For instance, if the database has a Users table, then a route can be written to find all of them, find one by its id, find by other criteria, create a user, delete a user, update a user
  • An example with only the findAll request
const router = require('express').Router()
const {User} = require('../db/models')
module.exports = router
router.get('/', async (req, res, next) => {
  try {
    const users = await User.findAll();
    res.json(users)
  } catch (err) {
    next(err)
  }
})

Socket.io

  • /server/socket
  • Socket methods are defined and executed
module.exports = io => {
  io.on('connection', socket => {
    console.log(`A socket connection to the server has been made: ${socket.id}`)
socket.on('disconnect', () => {
      console.log(`Connection ${socket.id} has left the building`)
    })
  })
}

Authentication

  • /server/auth
  • Middleware for logging in, signing up, logging out
  • Can have other forms of authentication (i.e. google)

FRONT END

Short story: React components and routing, Redux store, front end socket

Longer Story: Let’s start at:

Front End Entry Point

  • /client/index.js
  • Renders the application within a target html element, and passing the features below
  • Makes the redux store available through Provider, at any location in the front end
  • Adds front-end routing (and browser history) through Router from react-router-dom
  • Includes another React component, in this case App
  • An example
ReactDOM.render(
  <Provider store={store}>
    <Router history={history}>
      <App />
    </Router>
  </Provider>,
  document.getElementById('app')
)
  • Starting from this file, I find it good practice to sketch how all the components are connected, which ones are stateful, which ones dispatch actions to the redux store,
  • In this case, we have the front end entry point, index.js, which then renders app.js, which has navbar.js and routes.js
  • navbar.js, as its name, is the navbar element at the top of the page and has links to home, login, and signup
  • routes.js has a switch statement, which displays login.js by default, (or signup when clicked), and has a conditional that displays user-home.js if the user has logged in

Redux Store

  • First of all, redux is not required, but
  • Redux is a way to keep track of a global state and access it from any component, whether it is stateful or not.
  • A Redux store is the return value of a combination of all the reducers, usually a set of objects embedded into one object. In this example, The default value to an empty object until a request to the database is made.
  • Reducer function is a function that is invoked when a dispatch function is invoked. The reducer accepts as input the current state (which is either the entire store or a part of it) and an action object.
  • Action creator is a function that returns an object with at least two key-value pairs, one of which is the label/type of action to perform, the other is any sort of information that to pass in order to perform this type of action (i.e. to find a user by name, you’d pass that user’s name).
  • Thunk is “middleware that allows returning a function rather than an action” — github.com/reduxjs. This is useful when you’d like to make a request to the database before sending an action to your reducer
//action creator 
const getUser = userInfo => ({type: 'GET_USER', user: userInfo})
// thunk
const me = () => async dispatch => {
  try {
    const res = await axios.get('/auth/me')
    dispatch(getUser(res.data || defaultUser))
  } catch (err) {
    console.error(err)
  }
}
//reducer
const reducer = function(state = {}, action) {
  switch (action.type) {
    case 'GET_USER':
      return action.user
    default:
      return state
  }
}

What about /public ?

  • /public
  • Contains index.html, which contains the element into which /client/index.js will insert the front-end entry point.
  • Contains css files, bundle.js, bundle.js.map (these last two generated by webpack)
